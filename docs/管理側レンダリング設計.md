# 管理側レンダリング設計

## 方針

- 管理画面はSPAにする。
- Next.jsのページ遷移は使用しない。
- URLは使用する。
- データ取得はブラウザからSupabaseへ直接行う。
- 取得データはTanStack Queryのメモリキャッシュに保持する。
- 管理データ本体はIndexedDB、localStorage、sessionStorageに保存しない。
- 取得範囲設定はlocalStorageに保存する。
- 更新処理はServer Actionに統一する。

## URL

管理画面は `/admin` 配下で扱う。

```txt
/admin
/admin/products
/admin/products/new
/admin/products/<product_id>
/admin/orders
/admin/orders/<order_id>
/admin/news
/admin/news/new
/admin/news/<news_id>
/admin/inquiries
/admin/inquiries/<inquiry_id>
/admin/customers
/admin/customers/<customer_id>
/admin/settings
```

未定義のURLは管理画面内の404を表示する。

## ナビゲーション

- サイドバー操作ではNext.jsのページ遷移を発生させない。
- `history.pushState` でURLを更新し、SPA内部の表示画面を切り替える。
- ブラウザバック/進むは `popstate` で処理する。
- 検索、絞り込み、ソートはクライアント状態として保持し、URLに反映しない。

## データ取得

初回アクセス時は以下を行う。

```txt
1. ログイン状態と管理者権限を確認する
2. 管理用metaを取得する
3. 商品、カテゴリ、ブランドを全件取得する
4. 注文、お問い合わせ、顧客を設定された取得範囲で取得する
5. 取得したデータをTanStack Queryに保存する
```

画面切り替え時は、切り替え先のデータ種別だけmetaを確認する。

```txt
/admin/products
=> 商品metaを確認する

/admin/orders
=> 注文metaを確認する

/admin/customers
=> 顧客metaを確認する
```

meta確認後の処理:

```txt
1. そのデータのキャッシュがなければ取得する
2. removed_at が前回確認時より新しければ、そのデータを全件取得し直す
3. updated_at が前回確認時より新しければ、更新された行だけ取得してキャッシュに反映する
4. どちらも変わっていなければ何もしない
```

metaの日時はDB更新時にDB側の `now()` で保存する。クライアントはDBから返された日時をそのまま保持し、次回の同期判定に使う。ブラウザの現在時刻は使わない。

## 取得範囲

- 商品、カテゴリ、ブランドは全件取得する。
- 注文、お問い合わせ、顧客は取得範囲を指定して取得する。
- デフォルトの取得範囲は過去90日とする。
- 取得日数とオフセット日数は設定画面から変更できる。
- 取得範囲設定はlocalStorageに保存する。
- 注文、お問い合わせ、顧客の期間判定は `created_at` を基準にする。

範囲計算:

```txt
取得開始 = 現在時刻 - (オフセット日数 + 取得日数)
取得終了 = 現在時刻 - オフセット日数
```

例:

```txt
取得日数: 90
オフセット: 0
=> 直近90日

取得日数: 90
オフセット: 90
=> 90日前〜180日前
```

取得範囲を変更した場合は、注文・お問い合わせ・顧客のQuery Cacheを破棄して再取得する。

## キャッシュ

- TanStack Queryのメモリキャッシュだけを使う。
- 同じタブ内のSPA操作中はキャッシュを保持する。
- リロード、タブを閉じる、ブラウザ終了でキャッシュは消える。
- ログアウト時はQuery Cacheを破棄する。

## 検索と絞り込み

- 検索、絞り込み、ソートはキャッシュ済みデータに対して行う。
- 入力のたびにDBへ問い合わせない。

## 更新

- 読み取り、meta確認、差分取得、全件取得はブラウザからSupabaseへ直接行う。
- 作成、更新、削除、公開状態変更などの変更処理はServer Actionで行う。
- ブラウザからSupabaseへ直接 `insert` / `update` / `delete` しない。
- Server Action内で認証、認可、validation、競合チェック、関連テーブル更新、ISR再検証を行う。
- metaの日時更新はDB側の `now()` で行う。
- 複数のDB更新は1つの処理としてまとめ、途中失敗による不整合を防ぐ。

## 未保存フォーム

- 下書き永続化はしない。
- SPA内で未保存のまま別画面へ移動する場合は確認ダイアログを出す。
- リロード、タブ閉じ、ブラウザ終了には `beforeunload` で警告する。
- 破棄を選んだ場合、未保存内容は消える。

## 更新競合

- 編集ページを開く時は、対象データ種別のmetaを取得する。
- metaが前回確認時より新しければ、差分取得または全件取得でTanStack Queryのキャッシュを更新する。
- その後、TanStack Queryのデータからフォーム初期値を作る。
- フォームstateには、フォーム作成時点の `updated_at` を `originalUpdatedAt` として保持する。
- 保存時は `originalUpdatedAt` で競合チェックする。
- 保存時はServer Actionで `id` と `originalUpdatedAt` がDBの `updated_at` と一致する場合だけ更新する。
- 保存成功時はDB側で `updated_at` を `now()` に更新し、返却された最新データでTanStack Queryを更新する。
- `updated_at` が一致しない場合は競合として扱い、保存しない。
- 競合時は最新データを取得し、入力内容は保持したまま競合ダイアログを表示する。
- 競合ダイアログでは「最新データで開き直す」か「自分の入力で上書き保存する」かを選ばせる。
- 「最新データで開き直す」を選んだ場合は、自分の入力を破棄して最新データをフォームに反映する。
- 「自分の入力で上書き保存する」を選んだ場合は、競合時にDBから再取得した最新データの `updated_at` を使って再保存する。
- 上書き保存時も `id` と最新データの `updated_at` がDBの `updated_at` と一致する場合だけ更新する。
- 自動マージはしない。

## セキュリティ

- ブラウザにはSupabaseのpublishable/anon keyだけを置く。
- service role keyはブラウザに置かない。
- RLSで管理者だけが管理用データを読めるようにする。
- 画面に不要な機密情報は取得しない。
- 注文、顧客、決済関連データは永続ストレージに保存しない。

## 管理イベント

- 管理画面に通知したい出来事は `admin_events` に記録する。
- `admin_events` は管理者操作だけでなく、新規注文やお問い合わせなど管理画面に関係する出来事も扱う。
- 通知文は `event_type` と `resource_type` から決める。
- 商品、注文、お問い合わせなどの各テーブルを直接Realtime購読しない。

## Supabase Realtime

- Supabase RealtimeはBroadcastを使う。
- Broadcastは `admin_events` の追加を通知する用途に限定する。
- Supabase Realtimeで受け取った通知はSonnerで表示する。
- 通知クリックによる同期処理は実装しない。
- 通知を受けてもTanStack Queryのキャッシュは更新しない。
- データの最新化は、画面切り替え時、編集ページを開く時、再読み込み時のmeta確認で行う。
