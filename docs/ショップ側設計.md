# ショップ側設計

ショップ側の設計は、現在仮の状態です。
変更予定があるので、あまり参考にしないで。

## URL

```txt
/
/items
/items/<product_id>
/news
/news/<news_id>
/contact
/law
/privacy
```

商品一覧の絞り込み、検索、並び替え、ページ位置はクエリパラメータで表現する。

```txt
/items?category=<category_slug>
/items?brand=<brand_slug>
/items?status=sale
/items?q=<keyword>
/items?sort=new
/items?page=2
```

未定義のURLはショップ側の404を表示する。管理画面配下のURLはショップ側404ではなく、管理側ルーティングに任せる。

## ナビゲーション

- ショップ側はSEOを重視するため、Next.jsのページ遷移を使用する。
- トップ、商品一覧、商品詳細、お知らせ一覧、お知らせ詳細、ガイド系ページは通常のページとして扱う。
- ヘッダー、フッター、カートドロワー、お気に入りドロワーは共通UIとして配置する。
- 商品一覧の絞り込み、検索、並び替え、ページ位置はURLへ反映し、共有、戻る、進む、リロードで状態が復元されるようにする。
- カート、お気に入り、テーマ切り替えなど、SEOに不要な操作はClient Componentで扱う。

## レンダリングとキャッシュ

- ショップ側の主要ページはISRを基本とする。
- トップページは商品、お知らせ、カテゴリ、ブランドの公開情報を取得し、ISRで配信する。
- 商品一覧ページはSEOのため初期表示分をサーバー側で描画する。
- 商品詳細ページは公開中の商品を静的生成またはISRで配信する。
- お知らせ一覧、お知らせ詳細は公開済みのお知らせをISRで配信する。
- お問い合わせ、特定商取引法、プライバシーポリシーは静的またはISRで配信する。
- 商品在庫や価格など購入可否に関わる情報は、購入手続き開始時にサーバー側で最新データを確認する。
- 管理側で公開データを変更した場合、関連するショップ側パスを再検証する。

## データ取得

- 初期表示に必要な公開データは、ブラウザからSupabaseへ直接アクセスせず、サーバー側で取得する。
- 公開側で取得する商品は `active` のみとし、`draft` と `archived` は表示しない。
- 商品一覧、トップページの商品カードでは、一覧表示に必要な項目だけを取得する。
- 商品詳細では、商品説明、画像、スペック、配送表示、関連商品に必要な項目を取得する。
- カテゴリ、ブランドは公開中の商品に紐づくものを基本に表示する。
- お知らせは公開済みのものだけを取得し、公開日時の新しい順に表示する。
- 画像はR2などの外部ストレージに保存し、表示サイズに合った最適化済み画像を使用する。

## 取得範囲

- トップページの商品は、新着またはおすすめとして管理側で指定した件数のみ取得する。
- トップページのお知らせは最新3件を取得する。
- 商品一覧の初期表示件数は24件を基準にする。
- 商品一覧の追加読み込みはページネーションまたは「次へ」ボタンで行う。
- 商品一覧でSOLD OUTの商品を表示する場合、購入可能な商品より後ろに並べる。
- 商品詳細の関連商品は、同カテゴリ、同ブランド、公開中、対象商品以外を条件に取得する。
- お知らせ一覧の初期表示件数は10件を基準にする。

## フィルタとソートの挙動

- フィルタ、検索、ソートの状態はURLクエリを正とする。
- 初回表示とリロード時は、URLクエリをもとにサーバー側で表示内容を決定する。
- 画面操作時は、URLを更新しつつ同一取得範囲内ではクライアントで即時反映する。
- 未取得ページが必要な場合のみ、追加データを取得する。
- キーワード検索は商品名、ブランド名、カテゴリ名、商品説明の要約を対象にする。
- ソートは新着順、価格の安い順、価格の高い順を用意する。
- セール、NEW、SOLD OUTなどの状態表示は商品データから計算し、表示ロジックを共通化する。

## カートとお気に入り

- カートとお気に入りはユーザーログインを前提にせず、ブラウザのlocalStorageに保存する。
- カートに入れた時点では在庫を確保しない。
- カート内の数量は、表示中の商品データをもとに上限を制御する。
- 購入手続き開始時、サーバー側で最新の価格、在庫、販売状態を検証する。
- 検証結果がカート内容と異なる場合は、購入手続きへ進めず、変更点をカートに表示する。
- お気に入りは端末内の補助機能として扱い、サーバーには保存しない。

## 注文と決済

- 購入手続きはServer ActionまたはRoute Handler経由で開始する。
- サーバー側で認証不要の購入者情報、配送先、カート内容、価格、在庫、販売状態をvalidationする。
- validation後、注文の仮レコードを作成し、Stripe Checkoutなどの決済画面へ遷移する。
- 決済完了はWebhookを正とし、Webhook内で注文確定、在庫更新、メール送信、管理側通知に必要な処理を行う。
- 決済失敗、キャンセル、有効期限切れの場合は、注文を未確定のまま扱い、在庫は減らさない。
- 注文確定後の在庫更新と注文ステータス更新は1つの処理としてまとめ、途中失敗による不整合を防ぐ。

## お問い合わせ

- お問い合わせ送信はServer Actionを使用する。
- Server Action内でvalidation、スパム対策、Supabase保存、メール通知を行う。
- フォーム送信後は、送信完了画面または送信完了メッセージを表示する。
- 管理側で対応するため、お問い合わせ内容は管理側の一覧、詳細で参照できる形式で保存する。

## 未保存フォーム

- お問い合わせフォーム、購入者情報フォーム、配送先フォームは下書き永続化しない。
- 未送信のまま別画面へ移動する場合は確認ダイアログを出す。
- リロード、タブ閉じ、ブラウザ終了には `beforeunload` で警告する。
- 送信完了、決済画面への遷移、破棄を選んだ場合は未保存内容を消す。

## SEO

- トップ、商品一覧、商品詳細、お知らせ一覧、お知らせ詳細、ガイド系ページで個別のmetadataを設定する。
- 商品詳細は商品名、ブランド、価格、在庫状態、商品画像をOGPに反映する。
- お知らせ詳細はタイトル、公開日、ラベルをmetadataに反映する。
- 商品詳細にはProduct構造化データ、パンくずにはBreadcrumbList構造化データを設定する。
- 一覧ページのcanonicalはURLクエリの扱いを整理し、検索結果の重複評価を避ける。
- 非公開、削除、販売終了としてURLを残さない商品は404またはnoindexにする。

## UIとパフォーマンス

- スタイルの遅延を避けるため、初期表示に必要なCSSは通常のビルド成果物として読み込む。
- 色はCSS変数で管理し、`dark:` の多用を避ける。
- 画像、商品カード、ボタンなどはレイアウトシフトが起きないよう、寸法やアスペクト比を固定する。
- トップのヒーロー画像、商品詳細のメイン画像など、初期表示で重要な画像は優先度を上げる。
- 商品カード、価格表示、ステータスバッジ、ニュースラベル、ガイドリンクは共通コンポーネント化する。
- モバイルでは商品探索、カート確認、購入手続きへの導線を優先する。
- アニメーションは操作フィードバックに絞り、表示を待たせる演出は入れない。

## エラーと在庫差分

- 商品詳細で対象商品が存在しない、または公開対象外の場合はショップ側404を表示する。
- 商品一覧で該当商品がない場合は、条件変更を促す空状態を表示する。
- 購入手続き開始時に価格、在庫、販売状態が変わっていた場合は、変更内容を表示し、カート更新後に再操作させる。
- 決済画面から戻った場合は、成功、キャンセル、失敗をURLまたは注文状態から判定して表示する。
- 予期しないエラーはSentryなどに送信し、ユーザーには再試行可能な案内を表示する。
