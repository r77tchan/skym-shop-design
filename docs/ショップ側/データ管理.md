# データ管理（ショップ側）

SEO とユーザー体験の両立を目的とした、ショップ側のデータ取得・レンダリング方針。
URL 設計は `docs/ショップ側/URLとクエリ.md`、採用技術は `docs/技術スタック.md`、テーブル定義は `docs/DB設計.md` を参照。

## 方針の骨子

- 初回アクセス時は、商品データ入りの SSR HTML（CDN キャッシュ）を返す（SEO・非 JS・直リンク対応）。
- ハイドレーション後はクライアント SPA に切り替わり、URL は更新するがページ全体リロードはしない。
- 全商品（1000 件未満）をブラウザから Supabase へ直接取得し、IndexedDB にキャッシュする。
- 以降の一覧・詳細は IndexedDB から表示し、検索・絞り込み・並び替え・ページングはクライアントで完結する。
- データの鮮度は、リンク遷移ごとのチェックではなく、Supabase Realtime とブラウザ復帰時の差分同期で保つ。

## ルートごとのレンダリング設定

| ルート                                                                               | レンダリング                | index                | 補足                                                                                            |
| ------------------------------------------------------------------------------------ | --------------------------- | -------------------- | ----------------------------------------------------------------------------------------------- |
| `/`                                                                                  | SSR（CDN キャッシュ）       | index                | 商品 6 件＋お知らせ 3 件に依存                                                                  |
| `/items`、`/items?page=N`、`/items/<category_slug>`、`/items/<category_slug>?page=N` | SSR（CDN キャッシュ）       | ページングのみ index | 「一覧のページングと無限スクロール」参照。`?page=N` は別 URL ＝別キャッシュエントリ             |
| `/item/<product_id>`                                                                 | SSR（CDN キャッシュ）       | index                | 商品依存・動的ルート（初回 SSR → CDN キャッシュ）                                               |
| `/news`、`/news/<news_id>`                                                           | SSR（CDN キャッシュ）       | index                | news 依存                                                                                       |
| `/law`、`/privacy`                                                                   | prerender（SSG）            | index                | 完全静的・ビルド時確定                                                                          |
| `/contact`                                                                           | prerender（SSG）            | index                | 静的表示。送信は API route（`+server.ts`）へ POST（非 JS はネイティブ送信、JS は fetch で強化） |
| `/checkout`                                                                          | SSR                         | noindex              | 配送先・配送方法・備考入力。active な配送方法をサーバー側で取得し、注文作成と Stripe Session 作成はサーバー専用処理 |
| `/checkout/success`                                                                  | SSR                         | noindex              | `session_id` 依存・Stripe から注文取得（サーバー専用処理）                                      |
| `/checkout/cancel`                                                                   | prerender（SSG）            | noindex              | 静的メッセージ                                                                                  |
| 未定義 URL（404）                                                                    | prerender（SSG / fallback） | noindex              | ショップ側 404                                                                                  |

- **全ルート `ssr = true` / `csr = true`**（初回は SSR / prerender で HTML を焼き、ハイドレーション後は SPA 遷移）。`prerender` / SSR も `ssr = true` が前提。
- `csr = false` のページは作らない（共通ヘッダーのカート等で JS が要るため、`/law` 等もハイドレーションは必要）。
- prerender はビルド時生成（SSG・更新は再デプロイ）、SSR は初回アクセス時生成＋CDN キャッシュ（**時間ベース失効は使わず、イベント駆動のタグ無効化で再生成**）。無効化方針は「サーバーキャッシュの再生成（無効化）」を参照。
- **鮮度が要るルートは SSR（CDN キャッシュ）、完全に不変なルートは prerender（SSG）を選ぶ**。両者はルートごとの択一で、`export const prerender = true` のルートはビルド時に焼かれCDN キャッシュ無効化の対象外になる。

## レンダリングとデータ取得の状態

### 初回 / URL 直アクセス

- SSR（CDN キャッシュ）の HTML（= 正）をそのまま即時表示する。スケルトンは出さない。
- ハイドレーション後、バックグラウンドで以下を開始する。
  - 全商品データの取得と IndexedDB 構築
  - Supabase Realtime の購読
- IndexedDB 構築が完了したら、差分があればスケルトンを挟まず静かに最新へ更新する（差分がなければ見た目は変わらない）。
  - 一覧は無限スクロール対応のため、1 ページ目も IndexedDB 最新へ揃え、以降のページと鮮度を統一する。

### リンク遷移（クライアント SPA）

- リンクは押した瞬間に遷移し、URL を更新する。**遷移時に meta チェックはしない。**
- 表示データは IndexedDB から取得する（`liveQuery`）。
  - データがあれば即描画する。
  - まだ無い場合（IndexedDB 構築中など）は、商品カードだけスケルトン表示にし、データが入り次第自動で反映する。
  - ヘッダー・ナビ・フィルタなどのシェルは即描画する。

### load 関数の構成（universal load）

「初回は SSR で Supabase から、遷移後は IndexedDB から（サーバー往復ゼロ）」を 1 ルートで両立させるための制約。

- 一覧・詳細など両対応ルートは **`+page.ts`（universal load）で実装し、`+page.server.ts`（サーバー専用 load）は使わない**。サーバー専用 load はクライアント遷移時もサーバーへ data request を飛ばすため、「遷移＝往復ゼロ」が崩れる。
- universal load はサーバー（初回 SSR）とブラウザ（遷移）の両方で走る。`browser` で分岐し、**サーバー時のみ Supabase から取得して初回 HTML に焼く**。ブラウザ時は取得せず、表示は「読み出し」のとおりコンポーネント側の `liveQuery` / `stateQuery` に委ねる。
- CDN キャッシュ方針とタグも、このサーバー実行時に `setHeaders()` で付与する（「サーバーキャッシュの再生成（無効化）」参照）。`setHeaders` はブラウザでは no-op。

```ts
// +page.ts（+page.server.ts ではない点が肝）
import { browser } from '$app/environment'

export const load = async ({ params }) => {
  if (browser) return {} // 遷移時：取得しない（liveQuery が描画。サーバー往復ゼロ）
  return { initial: await fetchFromSupabase(params) } // 初回 SSR のみ
}
```

- サーバー専用処理が要る `/checkout` と `/checkout/success` だけ `+page.server.ts` を使う（「ルートごとのレンダリング設定」の表に従う）。

### 鮮度維持（概要）

- Supabase Realtime（カタログ用 meta テーブル購読）と、ブラウザ復帰時（`visibilitychange` / `focus` / `online`）の差分同期を併用する。
- Realtime（`postgres_changes`）は再接続中の変更を取りこぼすため、復帰時同期で補完する。Supabase クライアントは `realtime: { worker: true }` と `heartbeatCallback` を設定する。
- 更新が走っている間は、該当する商品カードをスケルトン表示にする。
- 仕組みの詳細は「カタログ用 meta」「差分同期と削除」を参照。

## IndexedDB データモデル

### 正規化ミラー

- `docs/DB設計.md` のテーブルを、そのまま IndexedDB に別ストアとしてミラーする（非正規化はしない）。
- 表示時に `liveQuery` 内で JOIN する。1000 件未満なので JOIN・メモリ処理のコストは無視でき、子テーブルが 1 件変わっても差分同期が素直なのが利点。
- 主キーは Supabase の `id` をそのまま inbound key に使う（`put` / `bulkPut` がそのまま upsert になる）。`id` は `bigint` だが 1000 件規模なら `number` 扱いで安全。
- IndexedDB には**公開商品（`is_published = true`）だけ**を保存する（非公開データはそもそもブラウザに届かない。「差分同期と削除」参照）。

### Dexie スキーマ

型は `docs/DB設計.md` の各テーブルをそのままミラーする。`stores()` に書くのは実際に where / sort するインデックスだけ（他カラムも保存はされる）。

```ts
db.version(1).stores({
  // 先頭 = 主キー（Supabase の id を inbound key に）
  products: 'id, sort_order, product_category_id, product_brand_id',
  product_images: 'id, product_id, [product_id+sort_order]',
  product_brands: 'id, slug, sort_order',
  product_categories: 'id, slug, sort_order',
  category_spec_definitions:
    'id, product_category_id, [product_category_id+sort_order]',
  category_spec_options:
    'id, category_spec_definition_id, [category_spec_definition_id+sort_order]',
  product_spec_values:
    'id, product_id, [product_id+category_spec_definition_id]',
  catalog_meta: 'id',
})
```

- 検索・SALE・在庫・スペック絞り込みはメモリ側で行うため、インデックスは最小限。

### 読み出し

- **一覧（検索・絞り込み・ソート・ページング）= 全件メモリ処理**。`products` 等を `liveQuery` で丸ごと取り、Svelte 側の `$derived` で filter / sort / slice する。URL や UI 状態に応じてリアクティブに切り替わる。
- **詳細 = インデックス引き**。`db.products.get(id)` ＋ 画像・スペックを複合インデックスで取得する。
- スケルトン判定は `stateQuery` の `current === undefined`（未取得）。`null` ＝ 本当に無い（404）、`[]` ＝ 空、と区別する。

## 一覧のページングと無限スクロール

### SEO / index

- index・CDN キャッシュ対象は**ページングのみ**：`/items`、`/items?page=N`、`/items/<category_slug>`、`/items/<category_slug>?page=N`。`?page=N` は別 URL ＝別キャッシュエントリとして page 単位でキャッシュされる（フィルタクエリはキャッシュ対象にしない＝エントリ爆発防止）。
- 各ページは**独立コンテンツ**：`page=N` は `(N-1)×12+1` 〜 `N×12` 件目だけを表示し、前のページは含まない（重複コンテンツを避ける）。
- `page=1` は canonical を `page` なしの URL に向ける。
- フィルタ付き URL（`brand` / `sale` / `keyword` / `<spec_slug>`）は **noindex ＋ canonical をフィルタなしの該当ページへ**。SSR はフィルタを適用せず素のページを返す。

### ページネーション実リンク（クローラー導線）

- リスト末尾に「さらに表示」を**実リンク `<a href="?page=N">`** として常設する（次ページがある限り）。
- 無限スクロールで JS 追加したコンテンツはクローラーに見えないため、各ページを独立した SSR HTML にし、実リンクで繋ぐことで page1 → 2 → 3 と全ページを辿れるようにする（Google 推奨の「無限スクロール＋ページネーション併用」）。非 JS フォールバックも兼ねる。
- 最終ページでは実リンクを「すべての商品を表示しました」に置換する。

### JS 有効時の挙動

- `IntersectionObserver` で末尾の「さらに表示」を監視し、ビューポートに入ったら自動ロード（手動クリックでも追加可）。URL は据え置き。
- `?page=N` で直接来た場合は N ページ目の位置（`(N-1)×12+1` 件目）から表示し、下方向へ無限スクロールする。1 ページ目には戻らない（各ページ独立を維持）。

### データ・ソート

- SSR：公開商品を `sort_order` 順（SOLD OUT は後ろ）で取得し、該当ページのスライスを焼く（フィルタ非適用）。
- JS 有効：IndexedDB（`liveQuery`）→ `$derived` でフィルタ・ソート・スライス。
- ページサイズは 12 件。canonical / robots（noindex）は `<svelte:head>` で出力する。

### フィルタ（URL クエリ）

- `keyword` / `brand`(slug) / `sale`(`true`) / `<spec_slug>`(slug) を URL に保持（リロード復元・共有可）。JS 有効時のみ IndexedDB 上で処理する。
- 変更時は `page` を削除して 1 ページ目に戻し、`replace` で履歴を汚さない。
- 非 JS ではフィルタは効かない（素のページが見える＝「フィルタは JS 必須」の線引きと一致）。

## カタログ用 meta

役割は「**変化検知**」と「**Realtime 購読対象**」。商品系（products / images / brands / categories / spec_definitions / spec_options / spec_values）の鮮度を表す軽量な 1 行テーブル。

```sql
create table catalog_meta (
  id smallint primary key default 1 check (id = 1),          -- 常に 1 行
  data_updated_at timestamptz not null default now(),        -- カタログ系の追加/変更の最終時刻
  data_reload_required_at timestamptz not null default now(), -- 全件再取得が必要になった最終時刻
  cache_schema_updated_at timestamptz not null default now() -- IndexedDB キャッシュ構造の更新時刻
);
```

- `data_updated_at` … カタログが増える / 変わる（新規公開・内容変更）の最終時刻。
- `data_reload_required_at` … 削除・非公開化など、差分取得では反映できず全件再取得が必要になった最終時刻。
- `cache_schema_updated_at` … ショップ側 IndexedDB のストア構造・インデックス・保存形式が変わった最終時刻。変わった場合は IndexedDB を削除して全件再構築する。
- 各テーブルの DB トリガで、INSERT/UPDATE 時は `data_updated_at`、DELETE・`is_published` が true→false の UPDATE 時は `data_reload_required_at` と `data_updated_at` を同じ `now()` に更新する（実装は管理/DB 側）。
- `cache_schema_updated_at` を更新する時は、`data_reload_required_at` と `data_updated_at` も同じ `now()` に更新する。
- Supabase から取得した `catalog_meta` のコピーを IndexedDB に保持し、差分取得時はローカル保存済みの `data_updated_at` より新しいレコードを取得する。

### 同期判定

```ts
async function syncCatalog(remoteMeta) {        // remoteMeta = Realtime payload.new or 取得
  const localMeta = await db.catalog_meta.get(1)

  // ① 初回 → 全件取得
  if (!localMeta) {
    await fullReload()
    await db.catalog_meta.put(remoteMeta)
    return
  }
  // ② IndexedDB の構造変更 → ローカルDBを削除して全件再構築（最優先）
  if (localMeta.cache_schema_updated_at !== remoteMeta.cache_schema_updated_at) {
    await resetCatalogDb()
    await fullReload()
    await db.catalog_meta.put(remoteMeta)
    return
  }
  // ③ 全件再取得が必要な変更 → 全件取り直し
  if (localMeta.data_reload_required_at !== remoteMeta.data_reload_required_at) {
    await fullReload()
    await db.catalog_meta.put(remoteMeta)
    return
  }
  // ④ 変わった → 全テーブルに差分クエリ（updated_at > ローカル meta の data_updated_at。多くは 0 件）
  if (localMeta.data_updated_at !== remoteMeta.data_updated_at) {
    const deltas = await Promise.all(
      TABLES.map((t) => fetchDelta(t, localMeta.data_updated_at)),
    ) // ネットワーク（並列）
    await db.transaction('rw', /* 全ストア + catalog_meta */, async () => {          // commit 1 回＝再評価 1 回
      for (const { table, rows } of deltas) if (rows.length) {
        await applyDelta(table, rows)
      }
      await db.catalog_meta.put(remoteMeta)
    })
  }
  // どちらも一致 → 何もしない
}
```

> 7 テーブルに毎回クエリが飛ぶが、変わっていないテーブルは 0 件で軽い。差分データの保存とローカル `catalog_meta` の更新は同じ IndexedDB トランザクションで行い、途中失敗時に取得済み扱いにならないようにする。

## 差分同期と削除

### 差分取得

- ローカル保存済み `catalog_meta.data_updated_at` より `updated_at` が新しいレコードのみ取得し、IndexedDB へ upsert（`bulkPut`、複数テーブルは 1 トランザクションにまとめる）。PostgREST の 1 リクエスト 1000 行上限は `.range()` で対応する。

### 子テーブル（画像・スペック値）

- 管理側で**子（画像 / spec 値）を変更したら、親 `products.updated_at` も bump** する（商品編集は画像・spec も含め 1 トランザクション＝ `docs/old/管理側設計.md` の「複数 DB 更新は 1 処理」と整合）。
- 差分で商品がヒットしたら、その商品の子を `product_id` で再取得し、IndexedDB 側はその商品の子を delete → 入れ直す（追加・削除・変更を丸ごと反映）。
- これにより `data_reload_required_at`（＝全件取り直し）は「商品/マスタ自体の削除・非公開化」だけで済み、子の小変更で全件取り直しにならない。

### 非公開・削除

- **非公開はショップ側にとって削除と同じ**。RLS で `anon` は `is_published = true` の商品しか SELECT できないようにし、非公開データはブラウザに一切届かないようにする。
- そのため非公開化・削除は差分（`updated_at > ローカル保存済み data_updated_at`）には現れない。**`data_reload_required_at` で検知し、全件取り直しで吸収**する。
- 全件取り直し＝公開商品を全件取得して IndexedDB を置き換える。消えた / 非公開になった商品は自然に IndexedDB から除去される。

## Realtime と復帰時同期

```ts
// Realtime：catalog_meta を購読 → 通知の payload.new で同期
supabase
  .channel('catalog-meta')
  .on(
    'postgres_changes',
    { event: 'UPDATE', schema: 'public', table: 'catalog_meta' },
    ({ new: meta }) => syncCatalog(meta),
  )
  .subscribe()

// 復帰時：取りこぼし補完
for (const ev of ['visibilitychange', 'focus', 'online'])
  addEventListener(ev, async () => syncCatalog(await fetchCatalogMeta()))
```

- 管理者が商品更新 → トリガで meta 更新 → Realtime 通知 → `syncCatalog` → `bulkPut` → `liveQuery` が画面反映、と一本に繋がる。

## サーバーキャッシュの再生成（無効化）

クローラー・非 JS・初回 HTML 向けのキャッシュ鮮度を、イベント駆動で保つ（JS ユーザーは IndexedDB ＋ Realtime で最新なので対象外）。

> **方式の決定**：ISR も選択肢にはなるが、本プロジェクトの規模・更新頻度では **通常 SSR ＋ CDN キャッシュ ＋ タグ ＋ Delete** で要件を満たせるため、まずはこちらを採用する。ISR は durable cache / request collapsing / cache shielding による同時アクセス耐性が必要になった場合に再検討する。

### 方針

- 一覧・詳細は **`+page.ts`（universal load）の SSR**。サーバー実行時（初回 / CDN ミス）に `setHeaders()` で CDN キャッシュ方針とタグを付与する。`setHeaders` はブラウザでは no-op なので、**クライアント遷移時は何もしない＝「load 関数の構成」の往復ゼロと両立**する。
- **時間ベース失効は使わない**。`Vercel-CDN-Cache-Control` を長期に設定し、実質「次の無効化まで」エッジにキャッシュさせる。
- 無効化は購入・管理変更などのイベント時に、**タグベースで該当キャッシュを Delete する**：
  - `invalidateByTag`（= stale-while-revalidate。無効化後 1 回目は古いものを返す）→ **使わない**。
  - **`dangerouslyDeleteByTag`（= 無効化後 1 回目を foreground で SSR し最新を返す。stale を返さない）→ こちらを使う**。

```ts
// +page.ts のサーバー実行時（初回 / CDN ミス時）に付与（ブラウザでは no-op）
setHeaders({
  'Vercel-CDN-Cache-Control': 'max-age=31536000', // エッジで長期キャッシュ（実質、無効化まで）
  'Vercel-Cache-Tag': 'items', // タグで束ねる（無効化単位）
})
```

### タグ設計

- 一覧系（`/`・`/items`・`?page=N`・`/items/<cat>`…）に共通タグ `items` を付与。商品が増減してページ数が変わっても 1 発で一括無効化できる（Vercel はパス指定パージが無く、タグで束ねる）。`?page=N` は別 URL ＝別キャッシュエントリだが、同じ `items` タグでまとめて消える。
- 商品詳細は個別タグ `product-<id>` ＋ `items`。
- news 系は `news`・`news-<id>`。
- タグ付与は `setHeaders` の `Vercel-Cache-Tag`（comma 区切り）で確実に行う。

### イベント → 無効化のマッピング

| イベント                                | 無効化するタグ                                                                                                       |
| --------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| 商品の作成 / 更新 / 公開・非公開 / 削除 | `items`（一覧・トップ）＋ `product-<id>`（該当詳細）                                                                 |
| news の作成 / 更新 / 削除               | `news` ＋ `news-<id>`                                                                                                |
| Checkout 開始時の在庫確保 / 期限切れ解除 | `items`（一覧・トップ）＋ `product-<id>`（該当詳細）                                                                 |

- 実際に `dangerouslyDeleteByTag()` を呼ぶのは管理側の更新処理、および Checkout 開始時の在庫確保・期限切れ解除処理。詳細実装は管理側に委ね、ショップ側はこのタグ設計・マッピングを前提にする。
- 商品更新時・Checkout 開始時の在庫確保・期限切れ解除時は、サーバーキャッシュ無効化（クローラー・非 JS・初回 HTML 向け）と、カタログ meta の bump（JS ユーザー向け Realtime/復帰時同期）の **2 系統**が走る。
- 商品詳細は関連商品にも依存するため、`items` タグも付与して商品系更新・在庫変動時にまとめて無効化する。

### 動作フロー

```
Checkout 在庫変動 or 管理変更
  ├─ カタログ meta を bump          → JS ユーザーが Realtime/復帰時に差分同期
  └─ dangerouslyDeleteByTag(...)    → 該当タグの CDN キャッシュを Delete
        ↓
無効化後の 1 回目アクセス → foreground で SSR（一瞬待つ）→ 最新を返す → CDN キャッシュ
        ↓
2 回目以降 → CDN から静的配信（次の無効化まで）。時間ベース失効なし
```

### トレードオフと採用前の検証

- CDN キャッシュは regional・best-effort（容量都合で evict されると次アクセスで再 SSR＝最新は返るが余計な生成）。request collapsing が無くアクセス集中時にスタンピードの可能性があるが、本プロジェクトの規模では実害は小さい。
- 採用前に、`dangerouslyDeleteByTag()` 後の 1 回目アクセスで **foreground 再生成され最新が返る**こと（stale を返さない）を実機で確認する。

### 補足

- 新商品の index 促進には動的 `sitemap.xml`（商品 / news / カテゴリの URL 一覧）も有効。SEO 全般として別途扱う。

## Progressive Enhancement の線引き

- 閲覧（トップ・商品一覧・カテゴリ・商品詳細・news・固定ページ）は JS 無効・直リンク・クローラーでも機能する。
- カート・お気に入りは localStorage を使うため JS 必須。検索・絞り込みも JS 必須。
