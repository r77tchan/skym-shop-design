# 管理側設計

## URL

```txt
/admin
/admin/products
/admin/products/new
/admin/products/<product_id>
/admin/orders
/admin/orders/<order_id>
/admin/news
/admin/news/new
/admin/news/<news_id>
/admin/inquiries
/admin/inquiries/<inquiry_id>
/admin/customers
/admin/customers/<customer_id>
/admin/settings
```

未定義のURLは管理画面内の404を表示する。

## ナビゲーション

- 管理画面は `/admin` 配下で、SPAにする。
- Next.jsのページ遷移を使用しないSPAだが、URLは更新する。

## データ取得

- データの取得はブラウザからSupabaseへ直接行う。
- Supabaseには管理用metaの、removed_at、updated_atが用意されている。
- 初回アクセス時、以下を取得し、IndexedDBに保存する。
  - 管理用meta
  - 商品、カテゴリ、ブランド全件
  - 注文、お問い合わせ、顧客の、一覧表示に使用している部分だけを全件
- 管理画面内ページ遷移時、管理画面内リロード時は、以下の手順でデータ更新を行う。
  - Supabaseから管理用metaを取得し、IndexedDBに保存されている既存の管理用metaと比較
  - removed_atが一致しない場合、全データ再取得
  - updated_atが一致しない場合、差分のみを取得
  - どちらも変わっていなければ何もしない
- 管理用metaの日時はSupabase側の `now()` で保存されており、IndexedDBはSupabaseから返された日時をそのまま保持し、次回の同期判定に使う。

## 取得範囲

- 商品、カテゴリ、ブランドは作成日に関わらず、全件取得する。
- 注文、お問い合わせ、顧客は、長期運用の際にデータ量が膨大になる恐れがあるため、期間を指定して取得する。
- デフォルトの取得範囲は過去90日。
- 取得日数とオフセット日数は管理画面の設定から変更できる。
- 期間判定は `updated_at` ではなく `created_at` を基準にする。
- 顧客データは注文、お問い合わせに紐づく顧客を取得する。
- 取得範囲を変更した場合は、注文、お問い合わせ、顧客の、一覧表示に使用している部分だけを全件再取得。

## フィルタとソートの挙動

- サーバーやSupabaseにアクセスせず、クライアントで完結させる。

## 更新

- 注文、お問い合わせ、顧客のデータは個人情報を含むため一覧画面では全データを取得していないので編集や詳細を開いた際に毎回データを取得する。
- 作成、更新、削除、などの変更処理はServer Actionを使用する。
- Server Action内で認証、認可、validation、競合チェック、関連テーブル更新、ISR再検証を行う。
- 更新リクエスト時、リクエスト側は、未変更含め編集前のデータを全て送り、サーバー側は、リクエスト側のupdated_atと最新Supabaseの対象データupdated_atを比較し、一致しない場合競合発生の為、更新は実施せずに、編集前のデータ、編集リクエストのデータ、最新Supabaseの対象データの3つをリクエスト側に返す。その後リクエスト側では、3つのデータを表示し、最新Supabaseの対象データを編集前のデータとして再リクエストを送るか、編集を破棄するか選択できる。
- metaの日時更新はDB側の `now()` で行う。
- 複数のDB更新は1つの処理としてまとめ、途中失敗による不整合を防ぐ。

## 未保存フォーム

- 下書き永続化はしない。
- 未保存のまま別画面へ移動する場合は確認ダイアログを出す。
- リロード、タブ閉じ、ブラウザ終了には `beforeunload` で警告する。
- 破棄を選んだ場合、未保存内容は消える。

## 通知

- 管理者による操作は、`admin_log` テーブルに記録している。
- このテーブルと注文テーブルを、Supabase Realtimeで購読する。
- 管理画面内では、更新通知をSonnerで表示する。
- 通知クリック処理や、通知発生時の自動更新処理は実装しない。
- 通知発生時、フラグを記録しておき、詳細ページ遷移時に古いデータが表示されないよう、必要に応じて賢く詳細表示の前に更新する処理を検討する。
